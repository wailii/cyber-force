# Cyber Force

这不是一个“更会说话”的 AI 助手项目。

先说明一点：

仓库里提到的 `主理人`，是对被服务对象的中性代称，不对应公开露出的真实姓名。

`Cyber Force` 也不是在讨论某种泛化的“外部力量”概念，而是在讨论一件更具体的事：

**如何围绕“主理人”这个具体的人，构建一个能代表其判断方式、价值排序、表达风格和记忆结构的数字分身。**

我们想开源的，不是一组提示词，也不是一套把大模型包起来的聊天壳子，而是一种更根本的产品思路：

**如果一个人想把自己的判断方式、价值排序、表达风格和长期记忆，逐步沉淀成一个可以长期调用、但又不失真的数字分身，这件事应该怎么设计？**

`Cyber Force` 试图回答的，就是这个问题。

## 我们想解决什么问题

今天大多数 AI 产品，解决的是“模型能不能更强”。

但对真实用户来说，更难的问题往往不是能力，而是：

- 这个系统到底是不是“像我”？
- 它是在替我放大判断，还是在悄悄改写我？
- 它记住的东西，是不是我真的愿意让它长期记住？
- 它在信息不够的时候，会不会开始瞎猜？
- 它能不能在需要的时候停下来，而不是为了显得聪明一路往下说？

我们认为，一个能长期陪伴人的 AI，不应该只追求流畅、讨喜、万能。

它更应该先解决三个基础问题：

1. **身份一致性**
   系统不能一会儿像你，一会儿像个通用客服。

2. **判断可托付**
   系统不是替你“热闹地回答”，而是替你守住判断边界。

3. **记忆可治理**
   记忆不是越多越好，而是越稳、越准、越可审计越好。

## 我们对产品的基本定义

`Cyber Force` 不是通用聊天机器人。

它更像一个“主权人格运行时”：

- 你可以和它对话
- 你可以让它帮你整理问题
- 你可以让它记住一些长期有效的事实
- 你可以让它调用外部能力

但在这一切之前，它首先要回答：

**“我现在是不是应该继续？”**

这意味着，这个产品的第一职责不是生成内容，而是治理行动。

## 整体产品设计思路

### 1. 先有“我”，再有“能力”

很多 AI 产品一上来就先接工具、接知识库、接工作流。

我们反过来。

我们先定义一个系统在代表谁：

- 他怎么看问题
- 他怎样做判断
- 他在冲突时优先保什么
- 他有哪些不能被轻易改写的原则
- 他在什么情况下必须停下来确认

只有这些东西先被描述清楚，后面的能力接入才有意义。

否则，系统只是一个会调用工具的模型，不是在代表主理人这个人。

### 2. 先治理，再执行

`Cyber Force` 的设计不是“收到输入 -> 立刻回答”。

而是先判断：

- 这句话是在表达情绪，还是在提出任务？
- 这是短期想法，还是值得长期记住的东西？
- 这是让我起草建议，还是直接替人做决定？
- 这件事有没有外部后果？
- 信息是不是足够？

也就是说，它不是把“回答”作为第一反应，而是把“判断是否该回答、该回答到哪一步”作为第一反应。

### 3. 记忆不是缓存，而是人格资产

我们不把记忆看成聊天上下文的延长。

我们把记忆看成一种会反过来塑造系统行为的资产。

所以记忆必须分层：

- 有些只是当天观察，不该永久保留
- 有些是候选判断，需要多次验证
- 有些才配进入长期记忆

这背后的核心思想是：

**不是所有说过的话都值得被记住，更不是所有被记住的东西都值得进入人格层。**

### 4. 系统必须允许“停下来”

这是我们最看重的一点。

很多 AI 产品把“不断接话”当成体验优势。

但在高风险场景里，这恰恰是问题。

`Cyber Force` 的产品逻辑里，停下来不是失败，而是一种能力：

- 信息不够，就追问最小必要问题
- 有冲突，就把冲突摊开
- 要越权，就停在草稿层
- 置信度不够，就不要装懂

我们认为，一个值得托付的系统，必须具备这种“主动不过界”的能力。

## 产品逻辑的设计思路

如果把它理解成一个产品，而不是一个模型封装，那么它的逻辑可以概括为五步：

### 1. 听懂这是什么类型的输入

不是所有输入都该被同样对待。

同一句话，可能是：

- 任务请求
- 情绪表达
- 记忆写入
- 记忆检索
- 普通闲聊
- 需要澄清的模糊问题

产品首先要识别“这句话属于哪一类”，而不是急着生成答案。

### 2. 判断这件事该推进到哪一层

不是所有问题都该直接推进到“执行”。

有些只应该停在：

- 解释层
- 提问层
- 草稿层
- 候选记忆层

只有一小部分，才应该进入：

- 长期记忆
- 真实行动
- 对外表达

### 3. 让系统给出明确位置，而不是模糊成功

很多 AI 产品喜欢返回一个“已完成”。

但真实世界里，“完成”是模糊的。

例如：

- 是记到临时日志里了，还是记到长期记忆里了？
- 是起草完成，还是已经代表用户发出去了？
- 是系统理解了，还是只是复述了？

所以我们的产品逻辑强调：

**系统反馈必须带位置感。**

也就是它不仅告诉你“做了”，还要告诉你“做到了哪一层”。

### 4. 人可以看得见、改得动、追得回

一个长期陪伴人的系统，不能把关键状态都藏在黑箱里。

所以我们坚持：

- 长期记忆要能被人直接阅读
- 日志要能看见系统最近记住了什么
- 冲突和失败不能被悄悄吞掉
- 用户必须能理解系统为什么这样判断

这不是单纯为了调试。

这是产品信任的一部分。

### 5. 产品要允许“不完美但可恢复”

现实里，任何外部系统都会失败。

所以一个成熟产品，不应该假设自己永远成功，而应该设计“失败之后怎么办”。

我们的思路是：

- 失败不能静默
- 失败要有痕迹
- 失败要能重试
- 失败之后用户要知道当前状态

这和“高可用”的技术话术不是一回事。

这是用户体验的一部分：

**不要让用户以为系统已经帮他处理好了，结果其实没有。**

## 我们的方法论

如果只用一句话概括，我们的方法论是：

**先定义边界，再放大能力。**

具体展开，有六条。

### 1. 主权优先

系统服务的是一个具体的人，而不是平均用户。

它不能为了更通用，慢慢把人磨平成一个“适合模型处理的用户画像”。

### 2. 原则先于风格

很多产品容易把“像不像你”理解成措辞像不像。

我们认为不是。

真正决定“是不是你”的，是：

- 原则排序
- 决策习惯
- 冲突处理方式
- 对风险和边界的态度

风格只是最后一层。

### 3. 记忆要慢，不要贪

我们不相信“用户说过一次，就应该永久记住”。

长期记忆应该是慢变量。

它需要被提炼、验证、观察，必要时甚至要被挑战。

### 4. 冲突必须显性化

如果系统发现：

- 新说法和旧原则冲突
- 当前任务和长期身份冲突
- 用户短期情绪和长期目标冲突

它不应该帮忙圆过去。

它应该把冲突摊开。

因为一个系统如果总在替你“维持表面顺滑”，最后一定会失真。

### 5. 不确定时先暂停

我们宁可让系统看起来没那么丝滑，也不想让它在不确定时瞎猜。

对很多产品来说，这是反直觉的。

但对“可托付”这件事来说，这是底线。

### 6. 人始终保有最后解释权

无论系统多像你，它都不该变成一个不能被质疑的黑箱。

用户应该始终能够：

- 修正它
- 质疑它
- 覆盖它
- 撤销它

系统的存在，是为了放大人的主权，不是替代人的主权。

## 为什么要把这套东西开源

因为我们觉得，这个方向不该只由少数公司在黑箱里定义。

“人格化 AI”“长期陪伴”“记忆代理”“数字分身”这些词，很容易被说成一些很轻巧的概念。

但真正重要的问题是：

- 什么能记？
- 什么不能记？
- 谁来定义一个系统是不是还“像你”？
- 当系统和你的长期原则冲突时，应该听谁的？
- 当系统失败时，用户怎么知道？

这些问题，本质上是产品问题、治理问题、方法论问题。

我们希望把这套思路开源，不是为了证明某个实现最好，而是为了把这些真正关键的问题摊到台面上。

## 这份 README 刻意不写什么

这份 README 刻意不展开技术实现、部署方式、环境变量、接口调用和工程细节。

因为我们现在更想公开的是：

- 这套东西为什么存在
- 它应该怎样被设计
- 它和普通 AI 助手的差别是什么
- 我们为什么坚持用这种方法去做

技术实现会变，产品形态也会变。

但如果这些底层设计思路不先讲清楚，后面的所有实现都容易跑偏。

## 如果你也在想类似的问题

如果你也在思考这些问题：

- 怎么做真正“像某个人”的 AI 系统
- 怎么处理 AI 的长期记忆
- 怎么让系统既有能力，又不过界
- 怎么把“判断方式”而不只是“说话风格”沉淀下来

那欢迎你把这里当成一个开放起点。

我们不觉得这套答案已经成熟。

但我们相信，这个问题值得被认真地、公开地、持续地讨论下去。
